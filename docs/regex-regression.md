# Регрессия поиска и замены

Короткий ручной чек-лист для поиска и замены, особенно после изменений в
`Scintilla`, `Lexilla`, `PCRE2`, `src/fbe/FBEview.cpp`, `src/fbe/mainfrm.cpp`,
`src/fbe/apputils.cpp`.

## Когда запускать

- после обновления `Scintilla` или `Lexilla`;
- после обновления `PCRE2` или кода вокруг regex backend режима `Дизайн`;
- после изменений в логике поиска и замены;
- перед публикацией release-кандидата.

## Автоматические проверки

Сначала прогоняем:

```powershell
.\tools\build\verify-release.ps1
```

Эта команда уже включает:

- проверку runtime-бинарников;
- `test-pcre2.ps1` для базового runtime-контурa `PCRE2` в режиме `Дизайн`, включая:
  - UTF-8 поиск по кириллице;
  - multiline-якоря;
  - регистронезависимый ASCII-поиск;
  - невалидный шаблон с ожидаемой compile-error;
- `test-pcre2-wrapper.ps1` для wrapper-логики режима `Дизайн`, включая:
  - одиночный и глобальный поиск;
  - группы захвата и `SubMatches`;
  - `FirstIndex` после перевода строки в UTF-8;
  - multiline-поиск с группой захвата;
  - zero-length regex без зацикливания;
- `test-pcre2-replace.ps1` для замены в режиме `Дизайн`, включая:
  - `\0`, `\1..\9`, `\+`;
  - `\U`, `\L`, `\T`, `\Q`;
  - базовые кириллические кейсы для регистровых преобразований;
  - глобальную zero-length замену;
  - multiline-замену по якорям строки;
- `test-scintilla.ps1` для regex-поиска и regex-замены в режиме `Код`, включая:
  - поиск со смещением до второго совпадения;
  - регистронезависимый поиск и замену;
  - XML-подобные фрагменты с тегами и группами захвата;
  - сценарии `not found`;
  - одиночную GUI-подобную замену через `SCI_TARGETFROMSELECTION`;
  - невалидный regex без падения `Scintilla`;
  - zero-length поиск и одиночную zero-length замену;
- проверку словарей орфографии.

На текущем этапе этот автоматический набор считается достаточным минимальным
baseline для завершённой ветки обновления `PCRE2` и `Scintilla`. Дальше его
можно расширять, но это уже не обязательное условие для того, чтобы считать
само обновление библиотек закрытым.

## Подготовка документа

При необходимости можно использовать готовый файл:

- [D:\Download\FBeditor\tools\tests\regex-regression.fb2](D:\Download\FBeditor\tools\tests\regex-regression.fb2)

Подготовить тестовый текст, где есть такие фрагменты:

- `(1) 1 (22) 22`
- `alpha beta ALPHA`
- `я`
- `привет`
- `иван`
- `ИВАН`
- `abcd`
- две соседние строки `alpha` / `beta` / `gamma` для проверки многострочного поиска
- XML-фрагмент с тегами в режиме `Код`

Использовать [FBE.exe](D:\Download\FBeditor\out\Release\FBE.exe).

## Ручная проверка: режим «Код»

1. Открыть документ и перейти в режим `Код`.
2. Включить поиск с флажком регулярных выражений.
3. Проверить шаблон `(\d+)`:
   должен находить только число без круглых скобок.
4. Проверить шаблон `\(\d+\)`:
   должен находить именно число в круглых скобках.
5. Проверить шаблон `\((\d+)\)`:
   должен находить число в круглых скобках с группой захвата.
6. Проверить замену по шаблону `\((\d+)\)` на `[\1]`:
   результат для `(1)` должен быть `[1]`.
   Этот сценарий уже однажды ловил регрессию в режиме `Код`, когда `\1` превращалось в пустую строку и получалось `[]`.
7. Проверить повторный поиск по строке `(1) 1 (22) 22`:
   нужно несколько раз нажать `Найти далее` и убедиться, что поиск последовательно проходит по совпадениям и не зацикливается.
8. Проверить регистронезависимый поиск по `alpha` / `ALPHA`.
9. Проверить, что неверный regex не валит приложение.
   Допустимо сообщение об ошибке поиска или отсутствие совпадений, но не падение программы.
   В текущей автоматической проверке `Scintilla` этот сценарий закреплён как `not found`.
10. Проверить zero-length regex по `.*?`:
   поиск не должен падать, а одиночная замена должна вести себя предсказуемо и не зависать.
11. Проверить, что после нескольких замен редактор остаётся рабочим.

## Ручная проверка: режим «Дизайн»

1. Перейти в режим `Дизайн`.
2. Проверить обычный поиск без regex.
3. Проверить regex-поиск по `привет` в тексте `я` + новая строка + `привет`:
   совпадение должно находиться, а смещение не должно "съезжать" из-за UTF-8.
4. Проверить regex `^(beta)$` на тексте:
   `alpha`, новая строка, `beta`, новая строка, `gamma`.
   В этом пункте под `multiline` имеется в виду поведение якорей `^` и `$` по строкам, а не отдельная галка в интерфейсе.
   Должна находиться только строка `beta`.
5. Проверить `Заменить` по `(\d+)` -> `[\1]`:
   для `(1)` результат должен быть `([1])`.
6. Проверить `Заменить все` по `(alpha)` -> `[\1]`:
   строка `alpha beta alpha` должна превратиться в `[alpha] beta [alpha]`.
7. Проверить `Заменить` по `((ab)(cd))` -> `<\+>`:
   строка `abcd` должна превратиться в `<cd>`.
8. Проверить `Заменить` по `(a)?b` -> `X\1Y` на строке `b`:
   результат должен быть `XY`, без мусора вроде `X\1Y`.
9. Проверить `Заменить` по `(иван)` -> `\U\1\Q`:
   `иван` должен превратиться в `ИВАН`.
10. Проверить `Заменить` по `(ИВАН)` -> `\L\1\Q`:
    `ИВАН` должен превратиться в `иван`.
11. Проверить `Заменить` по `(ivan)` -> `\U\1\Q-\1`:
    `ivan` должен превратиться в `IVAN-ivan`.
12. Проверить `Заменить все` по `.*?` -> `#` на строке `ab`:
    результат должен быть `#a#b#`, без зависания и бесконечного цикла.
13. Проверить `Заменить все` по `^beta$` -> `[\0]`:
    `alpha`, новая строка, `beta`, новая строка, `gamma`
    должно превратиться в `alpha`, новая строка, `[beta]`, новая строка, `gamma`.
    Здесь тоже проверяется multiline-поведение якорей `^` и `$` по строкам.
14. Проверить, что после нескольких замен не ломаются выделение, фокус и `Undo`.
15. Проверить, что многострочный или сложный regex не вызывает падения.

## Что считать регрессией

Нужно останавливать выпуск и разбираться, если есть хотя бы одно из этого:

- `(\d+)` и `\(\d+\)` дают перепутанный результат в режиме `Код`;
- замена по `\1` работает неверно;
- поиск в режиме `Код` и `Дизайн` начал падать;
- regex в режиме `Дизайн` перестал работать после обновления `PCRE2`;
- некорректный шаблон валит приложение вместо сообщения об ошибке;
- `Заменить все` пропускает совпадения или зацикливается;
- одиночная regex-замена в режиме `Код` снова теряет группы захвата и превращает `[\1]` в `[]`;
- `FirstIndex` или фактическое место замены "съезжают" на кириллице или после перевода строки;
- `\Q`, `\+` или пустая `\1` в замене начинают работать не так, как описано в
  [docs/pcre2-compatibility.md](D:\Download\FBeditor\docs\pcre2-compatibility.md).

## Если проблема повторилась

1. Зафиксировать, в каком режиме это произошло: `Код` или `Дизайн`.
2. Сохранить точный шаблон поиска и строку замены.
3. Для режима `Код` перепроверить `tools/tests/test-scintilla.ps1`.
4. Для режима `Дизайн` посмотреть изменения в `src/fbe/RegexBackend*.cpp` и `src/fbe/FBEview.cpp`.
5. Если есть падение, повторить сценарий под `Visual Studio` и сохранить `Call Stack`.
