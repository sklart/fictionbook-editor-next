# Анализ reference-форка `evpobr/fictionbookeditor`

Эта заметка фиксирует результаты сверки с upstream/reference-репозиторием
`https://github.com/evpobr/fictionbookeditor`.

Цель сверки — не механически подтягивать старые изменения, а понять, какие
идеи из форка всё ещё полезны для текущей ветки модернизации FBE.

## Базовый вывод

На момент проверки 21 июня 2026 года reference-форк не является активным
современным upstream в обычном смысле:

- последний верхнеуровневый коммит в `master` датирован 3 декабря 2017 года;
- основная масса практически полезных изменений приходится на 2016–2017 годы;
- часть этих идей уже давно реализована в нашей текущей кодовой базе;
- оставшееся полезно в основном как исторический reference, а не как источник
  прямого cherry-pick.

## Что уже фактически учтено у нас

### 1. `FBV` на `MSXML6`

В reference-форке есть коммиты:

- `cfc15d9` — `Migrate FBV to MSXML 6.0 #205`
- `6224536` — `Refactor FBV using ATL & WTL`

Текущее состояние нашей ветки:

- `src/fbv/stdafx.h` уже подключает `msxml6.h`;
- `src/fbv` уже импортирован из reference-форка;
- поверх import уже внесены наши локальные доработки устойчивости и MUI-ресурсы
  для shell-команды `Validate`.

Вывод:

- заново подтягивать `FBV` из `evpobr` не нужно;
- reference-форк полезен только как историческая точка сравнения по отдельным
  локальным refactoring-идеям.

### 2. `ExportHTML` на `MSXML6`

В reference-форке есть коммит:

- `d0c9e8f` — `Migrate ExportHTML to MSXML 6.0 #205`

Текущее состояние нашей ветки:

- `src/export-html/StdAfx.h` уже подключает `msxml6.h`;
- `src/export-html/Utils.cpp` уже использует `Msxml2.DOMDocument.6.0` /
  `Msxml2.FreeThreadedDOMDocument.6.0`.

Вывод:

- этот practical-upstream уже фактически учтён;
- дополнительного переноса не требуется.

### 3. `MS Shell Dlg` в `FBV` и `ExportHTML`

В reference-форке есть коммиты:

- `cf9223e` — `Use MS Shell Dlg in FBV #214`
- `fbebdda` — `Use Ms Shell Dlg in ExportHTML plugin #214`

Текущее состояние нашей ветки:

- в `src/fbv/FBV.rc` диалог уже использует `FONT 8, "MS Shell Dlg", ...`;
- в `src/export-html/ExportHTML.rc` диалог уже использует
  `FONT 8, "MS Shell Dlg", ...`.

Вывод:

- эта часть reference-форка уже учтена;
- дополнительных действий не требуется.

### 4. LZMA-сжатие в установщике

В reference-форке есть коммит:

- `f0a90ac` — `Use LZMA compression in installer`

Текущее состояние нашей ветки:

- текущий NSIS-контур уже использует `lzma`;
- это поведение отражено и в скриптах, и в фактической сборке setup.

Вывод:

- practical-value для переноса уже отсутствует, потому что решение давно
  перекрыто нашей веткой.

## Что полезно только как идея, но не как готовый patch

### 1. Полная MUI-локализация приложения

В reference-форке есть коммит:

- `6a73ac9` — `Add Localization with MUI technology`

Практический смысл для нашей ветки:

- интересен как reference по MUI-подходу;
- уже частично отразился у нас в локализуемом shell-verb `Validate` через
  `FBVVerbResources.dll` и `*.mui`.

Почему не переносить прямо сейчас:

- у нас уже есть собственная текущая архитектура локализаций;
- задача полной MUI-миграции всего FBE не является текущим приоритетом;
- прямой перенос коммита конфликтовал бы с нашей современной структурой
  `src/runtime/packaging/docs` и с уже изменённым release-контуром.

Вывод:

- использовать только как справочный материал, не как прямой кандидат на
  импорт.

### 2. Cleanup предупреждений и SDL-related fixes

В reference-форке есть коммиты:

- `8cf910b` — `Enable more warnings`
- `d1a9cc0` — `Fix warnings preventing build with SDL checks`

Практический смысл для нашей ветки:

- полезны как каталог исторически проблемных мест;
- могут подсказать отдельные локальные правки при будущей модернизации.

Почему не переносить напрямую:

- структура проектов, toolset и зависимостей у нас уже сильно другие;
- часть файлов уже перемещена или переписана;
- прямой patch-трансфер даст больше конфликтов, чем пользы.

Вывод:

- держать в уме как reference при следующей локальной cleanup-модернизации
  `src/fbe`, но не включать в текущий обязательный план.

### 3. Улучшения диалогов и DPI/Vista UI

В reference-форке есть коммиты:

- `7986f81` — `Fix settings dialog issues in High DPI mode #214`
- `27f9ef5` — `Use themed file open and save dialogs`
- `043f55d` — `Add Vista file open/save dialogs`
- `7336b77` — `Use task dialog instead of message boxes closes #216`

Практический смысл для нашей ветки:

- это потенциально полезный набор UX/UI-идей для отдельной модернизации FBE;
- особенно интересны High DPI и точечные modern-dialog улучшения.

Почему не переносить прямо сейчас:

- тема не входит в текущий приоритет shell / installer / property handler /
  thumbnail / FBV;
- часть изменений затрагивает старые классы и диалоги, которые у нас уже
  разошлись с reference-кодом.

Вывод:

- это отдельный будущий исследовательский блок, а не текущая задача.

## Что не нужно переносить по проектным решениям

### 1. Legacy `FBSHell` thumbnail handler

В reference-форке есть коммит:

- `24be31b` — `Add thumbnail handler`

Почему это не наш путь:

- он относится к старому `FBSHell`-контуру;
- проектное решение нашей ветки — не возвращать legacy shell в стандартный
  runtime;
- текущий modern thumbnail provider уже реализован и отдельно подтверждён
  автоматическими и ручными проверками.

Вывод:

- не переносить;
- использовать только как исторический reference, если понадобится понять,
  как что-то делалось в старом shell-контуре.

### 2. Исторические installer/build-решения на `NuGet` / `Vcpkg`

В reference-форке есть коммит:

- `82f391e` — `Replace NuGet with Vcpkg closes #217`

Почему это не наш путь:

- наша ветка уже живёт в другой структуре зависимостей;
- у нас есть собственные `third_party`, build-скрипты и update-контур
  зависимостей;
- переход на `vcpkg` сейчас не является частью принятой архитектуры проекта.

Вывод:

- не переносить.

## Практический итог для нашей ветки

По результатам сверки reference-форк `evpobr/fictionbookeditor` сейчас даёт
нам не набор срочных patch-кандидатов, а в основном:

1. подтверждение, что import `FBV` в `src/fbv` был правильным решением;
2. подтверждение, что `MSXML6` и `MS Shell Dlg` в `FBV` / `ExportHTML`
   у нас уже учтены;
3. исторический reference по возможной будущей модернизации UI-диалогов,
   High DPI и локальных warning-cleanup работ;
4. дополнительное подтверждение, что legacy `FBSHell` назад в runtime
   возвращать не нужно.

## Что имеет смысл сделать дальше

Не пытаться тащить коммиты из `evpobr` механически.

Вместо этого:

- продолжать текущую ветку по собственному roadmap;
- при будущей модернизации `src/fbv` и UI FBE при необходимости точечно
  открывать отдельные reference-коммиты `evpobr`;
- отдельно вернуться к UX-блоку диалогов и High DPI только после закрытия
  более приоритетных shell/installer задач.
