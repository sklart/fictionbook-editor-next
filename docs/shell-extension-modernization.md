# Модернизация shell extension

Краткая памятка по текущему состоянию `FBShell.dll` и по тому, что с ним стоит
делать дальше.

## Что входит в `src\fbshell`

Важно: в стандартную сборку и стандартную поставку теперь входит именно modern
property handler. Legacy-компоненты (`ColumnProvider`, `ContextMenu`, `IconExtractor`) больше не подключены к активному проекту `FBShell`; они сохранены в дереве исходников только как исторический reference-контур.

Исторически в `src\fbshell` были собраны сразу три разных shell-компонента:

- `ColumnProvider` — старый провайдер колонок Проводника;
- `IconExtractor` — извлечение обложки FB2 для значков и превью-иконок;
- `ContextMenu` — пункт контекстного меню для запуска `FBV.exe`.

Это видно по регистрации трёх COM-классов в:

- `src\fbshell\FBShell.cpp`;
- `src\fbshell\ColumnProvider.rgs`;
- `src\fbshell\IconExtractor.rgs`;
- `src\fbshell\ContextMenu.rgs`.

Важно: эти `.rgs` больше не являются описанием актуальной поставки FBE.
Сейчас это только исторические reference-файлы, которые полезны для разбора
старой архитектуры и сравнения с modern-контуром, но не должны считаться
рабочим способом регистрации текущей shell-интеграции.

## Что выглядит устаревшим

### 1. `ColumnProvider`

`ColumnProvider` использует интерфейс `IColumnProvider` и регистрируется через
`Folder\shellex\ColumnHandlers`.

Это legacy-механизм старых версий Explorer. На современных Windows он выглядит
наиболее сомнительной частью `FBShell.dll`, потому что:

- привязан к старому API колонок Проводника;
- добавляет системную интеграцию, которой многие пользователи уже не
  пользуются;
- даёт дополнительную поверхность для проблем внутри `explorer.exe`.

Практический вывод: это первый кандидат на отключение или вынос из основной
поставки.

Текущий статус после модернизации:

- `ColumnProvider` больше не регистрируется в стандартной поставке по
  умолчанию;
- исходники и `.rgs` сохранены в проекте как legacy-контур;
- старые строки `InfoTip`, `TileInfo`, `Details` из `ColumnProvider.rgs`
  больше не являются источником истины для текущей поставки: их современный
  эквивалент теперь задаётся напрямую инсталлятором через `FictionBook.2`;
- в активном проекте отдельного build-флага для его возврата больше нет:
  для реанимации этого контура нужна будет уже отдельная осознанная сборочная
  ветка, а не случайное включение старой опции.

### 2. `ContextMenu`

`ContextMenu` добавляет пункт `Validate` для `.fb2`, папок и фона каталога, а
затем запускает `FBV.exe` через `CreateProcess`.

Функция сама по себе полезная, но реализация тоже legacy:

- регистрация идёт как shell extension, а не как современная отдельная команда
  приложения;
- обработчик живёт внутри процесса Проводника;
- всё поведение завязано на наличие `FBV.exe` рядом с DLL.

Практический вывод: это можно оставить дольше, чем `ColumnProvider`, но лучше
рассматривать как отдельный опциональный компонент, а не как обязательную часть
ядра FBE.

Текущий статус после модернизации:

- `ContextMenu` тоже больше не регистрируется в стандартной поставке по
  умолчанию;
- исходники и `.rgs` сохранены в проекте как legacy-контур;
- основной пользовательский сценарий `Validate` уже вынесен в обычную
  shell-команду без DLL, поэтому для возврата старого COM-обработчика тоже
  потребуется отдельная осознанная сборочная ветка.

### 3. `IconExtractor`

`IconExtractor` выглядит самой понятной и полезной частью старого пакета:

- он читает `coverpage` из FB2 через SAX;
- декодирует встроенный `binary` base64;
- строит bitmap для Проводника.

Но и тут есть нюансы:

- код работает внутри `explorer.exe`;
- внутри проекта живут очень старые встроенные копии:
  - `zlib 1.1.4` (`src\fbshell\zlib\zlib.h`);
  - `libpng 1.2.8` (`src\fbshell\libpng\png.h`);
  - `libjpeg 6b` (`src\fbshell\libjpeg\jversion.h`);
- любые ошибки в парсинге FB2 или декодировании картинки бьют уже по оболочке
  Windows, а не только по FBE.

Практический вывод: эту часть имеет смысл сохранить последней, если вообще
сохранять shell integration.

Текущий статус после модернизации:

- legacy `IconExtractor` и весь старый shell-контур выведены из стандартной
  поставки;
- для `.fb2` оставлена обычная иконка файла через `DefaultIcon` от `FBE.exe`;
- сценарий `cover extraction` отложен как отдельная будущая задача, если к
  нему вообще будет смысл возвращаться;
- это уменьшает риск для `explorer.exe`, пока modern property handler уже
  закрывает сценарии колонок, tooltip и правой панели сведений Windows 11
  без обязательной зависимости от старого shell-контура.

## Отдельный риск регистрации

`*.rgs` файлы до сих пор пишут не только в `HKCR`, но и в
`HKLM\Software\Microsoft\Windows\CurrentVersion\Shell Extensions\Approved`.

При этом текущий установщик FBE работает в per-user режиме. Значит, shell
extension сейчас находится на неудобной границе между:

- современной установкой без повышения прав;
- исторической COM-регистрацией, рассчитанной на более старую модель системы.

Это нужно отдельно проверить вручную на целевой Windows:

- проходит ли регистрация полностью;
- нужен ли реально `Approved` в `HKLM`;
- работает ли сценарий без администраторских прав;
- не остаётся ли у пользователя частично зарегистрированное состояние.

## Рекомендуемый порядок модернизации

1. Зафиксировать фактическое поведение `FBShell.dll` на Windows 10/11:
   - значки FB2;
   - всплывающая подсказка;
   - пункт `Validate`;
   - дополнительные колонки.
2. Разделить в голове и в упаковке три функции:
   - значки;
   - контекстное меню;
   - колонки.
3. Первым кандидатом на отключение сделать `ColumnProvider`.
4. Затем решить судьбу `ContextMenu`: оставлять ли его только как legacy-опцию
   или заменить на менее инвазивный способ запуска `FBV`.
5. Отдельно решить, сохраняем ли `IconExtractor` и нужно ли обновлять его
   bundled-графические библиотеки.

## Практическая рекомендация на сейчас

Legacy-контуры shell extension по-прежнему стоит считать зоной повышенного
риска, потому что они живут внутри `explorer.exe` и завязаны на старые COM-
сценарии.

Для обычной работы FBE они не нужны. Актуальная ценность `FBShell.dll` сейчас
сводится к modern property handler и удобствам Проводника, а не к работе
редактора как таковой.

## Текущий состав стандартной поставки

На текущем этапе стандартная shell-поставка FBE считается такой:

- modern property handler:
  - `FBShell.dll` для `Win32 Explorer`;
  - `FBShell64.dll` для `64-bit Explorer`;
- обычная файловая иконка `.fb2` через `DefaultIcon` от `FBE.exe`;
- обычная shell-команда `Validate`, запускающая `FBV.exe` без COM-DLL;
- shell-строки `InfoTip`, `TileInfo`, `Details`, `PreviewDetails` для
  `FictionBook.2`;
- published FBE-specific schema для книжных полей:
  `FBE.Genre`, `FBE.Sequence`, `FBE.DocumentVersion`,
  `FBE.DocumentDate`, `FBE.Keywords`, `FBE.DocumentId`.

Что в стандартную поставку больше не входит:

- legacy `ColumnProvider`;
- legacy `ContextMenu`;
- legacy `IconExtractor`;
- старый bundled image-stack (`zlib/libpng/libjpeg`) внутри активной сборки
  `FBShell`.

## Результат ручной проверки

Ручная проверка уже обновлённой сборки на современной Windows показала
следующее:

- значения `.fb2` отображаются в колонках Проводника;
- modern property handler работает стабильно;
- FBE-specific поля `Keywords`, `Genre`, `Sequence`, `DocumentId`,
  `DocumentVersion`, `DocumentDate` подтверждены на реальной машине через
  published property schema;
- после полной shell-регистрации tooltip Windows 11 показывает не только
  базовые поля, но и данные из `.fb2`;
- правая панель сведений Windows 11 тоже начинает показывать данные из `.fb2`,
  если помимо `Details` прописан отдельный ключ `PreviewDetails`;
- legacy shell-контур для сценария колонок больше не нужен;
- активный проект `FBShell` теперь не включает `ColumnProvider`, `ContextMenu` и `IconExtractor` вообще; modern DLL собирается только вокруг property handler.
- пункт `Validate` больше не зависит от legacy `ContextMenu`, потому что
  вынесен в обычную shell-команду;
- для `.fb2` используется обычная иконка файла через `DefaultIcon`.

Практический вывод: полезные сценарии Проводника теперь закрываются modern
property handler, FBE-specific property schema для серии книги и обычной
shell-командой `Validate`, без обязательной зависимости от legacy
`FBShell`-контуров.

Уточнение по правой панели сведений:

- одного ключа `Details` для современной Windows 11 оказалось недостаточно;
- рабочим оказался вариант, когда для `FictionBook.2` дополнительно
  прописывается `PreviewDetails`;
- после этого правая панель сведений начала показывать значения из `.fb2`,
  включая `Авторы`, `Название`, `Язык`, `FBE:Жанр`, `FBE:Серия`,
  `FBE:Версия документа`, `FBE:Дата документа`, `FBE:Ключевые слова`,
  `FBE:Идентификатор документа`.

Дополнение по источникам регистрации:

- актуальная поставка должна смотреть на `MakeInstaller.nsi`, PowerShell-
  скрипты регистрации property schema и modern property handler;
- legacy `.rgs`-файлы нужно читать только как исторический reference, а не как
  описание того, что реально должно попасть в систему сегодня.
