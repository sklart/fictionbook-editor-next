# План переработки FBShell.dll

Этот документ описывает практический план модернизации shell-интеграции FBE под
современные версии Windows и отдельно выделяет направление для современных
колонок `.fb2` в Проводнике.

## 1. Цели

Нужно разделить две разные задачи:

1. привести shell-интеграцию FBE в предсказуемое состояние на Windows 10/11;
2. отдельно спроектировать современное отображение метаданных `.fb2` в
   Проводнике.

Важно не пытаться "спасти всё сразу" внутри текущего `FBShell.dll`.
Исторический код нужно сначала разложить по функциям и только потом решать,
что сохраняем, что переписываем, а что убираем.

## 2. Текущее состояние

Исторически в `src\fbshell` были смешаны три разных сценария:

- `ColumnProvider` — старые колонки Проводника;
- `IconExtractor` — значки и извлечение обложки;
- `ContextMenu` — пункт `Validate`.

Ручная проверка на современной Windows уже показала:

- обычная ассоциация `.fb2` работает;
- modern property handler стабильно показывает метаданные `.fb2` в колонках
  Проводника;
- после полной shell-регистрации tooltip Windows 11 показывает данные из
  `.fb2`, а правая панель сведений тоже начинает их показывать, если для
  `FictionBook.2` дополнительно прописан `PreviewDetails`;
- старый механизм колонок больше не нужен как базовый путь.

## 3. Принцип переработки

Новый контур стоит строить по правилу:

- отдельно значки;
- отдельно команды оболочки;
- отдельно свойства и колонки;
- никакой скрытой зависимости одной функции от другой.

То есть целевое состояние — не один "комбайн" с историческими COM-классами, а
понятные независимые компоненты, которые можно включать, тестировать и
поставлять по отдельности.

## 4. Потоки работ

### Поток A. Инвентаризация и декомпозиция FBShell.dll

Цель: подготовить почву для безопасной переработки.

Шаги:

1. Зафиксировать текущие классы и их регистрацию:
   - `ColumnProvider`;
   - `IconExtractor`;
   - `ContextMenu`.
2. Разделить в исходниках и документации их роли и зависимости.
3. Проверить, какие части реально нужны пользователям сегодня.
4. Подготовить новый состав поставки:
   - обязательные компоненты;
   - optional legacy-компоненты;
   - experimental-компоненты.

Результат этапа:

- понятная матрица "функция -> код -> регистрация -> способ установки".

### Поток B. Очистка legacy-частей

Цель: убрать то, что почти наверняка не нужно поддерживать дальше.

Приоритет:

1. `ColumnProvider` — первый кандидат на вывод из стандартной поставки.
2. HKLM-ориентированные legacy-хвосты регистрации.
3. всё, что не работает в per-user установке и не даёт реальной пользы.

Результат этапа:

- уменьшение риска для `explorer.exe`;
- более честная и компактная shell-интеграция.

Промежуточный фактический статус:

- `ColumnProvider` уже выведен из стандартной регистрации по умолчанию;
- `ContextMenu` уже выведен из стандартной регистрации по умолчанию;
- `IconExtractor` уже выведен из стандартной сборки по умолчанию;
- legacy-код и `.rgs` оставлены в дереве исходников, но уже отключены от активного проекта `FBShell`;
- их нужно рассматривать как reference-слой для анализа старой архитектуры, а
  не как живую спецификацию текущего инсталлятора.

### Поток C. Современное контекстное меню

Цель: вернуть полезный сценарий `Validate` в современном и поддерживаемом виде.

Варианты:

1. минимальный: сохранить COM-based context menu, но сделать его отдельным
   optional-компонентом с нормальной диагностикой и перерегистрацией;
2. предпочтительный: изучить менее инвазивный современный способ интеграции
   команды запуска `FBV.exe` для `.fb2`.

Минимальный MVP:

- пункт `Validate` для `.fb2`;
- корректный запуск `FBV.exe`;
- отсутствие падений `explorer.exe` при ошибках;
- понятное поведение в per-user установке.

Промежуточный фактический статус:

- legacy `ContextMenu` уже выведен из стандартной регистрации по умолчанию;
- код и `.rgs` сохранены в проекте;
- основной сценарий `Validate` уже закрыт обычной shell-командой без DLL;
- возврат старого COM-контурa теперь имеет смысл только как отдельная
  осознанная legacy-ветка, а не как штатная опция активного проекта.

### Поток D. Значки и обложки

Цель: понять, стоит ли сохранять shell-извлечение обложки из FB2.

Шаги:

1. проверить, нужен ли пользователям именно cover-based icon extractor;
2. оценить риск старого кода `zlib/libpng/libjpeg` внутри `src\fbshell`;
3. решить, переписывать ли этот контур или оставить только обычную иконку от
   `FBE.exe`.

Промежуточный фактический статус:

- внутри `IconExtractor` до сих пор используются встроенные библиотеки
  `zlib 1.1.4`, `libpng 1.2.8` и `libjpeg 6b`;
- стандартная сборка `FBShell` больше не подтягивает этот контур и его bundled image-stack вовсе; legacy-часть отключена от активного проекта.
- это усиливает аргумент в пользу отдельной ревизии `IconExtractor`, а не
  безусловного сохранения его в стандартной поставке.

Результат этапа:

- либо упрощённая и надёжная схема с обычной иконкой;
- либо отдельный modernized icon/thumbnail contour.

Следующий зафиксированный шаг по этому направлению теперь вынесен отдельно в:

- `docs/thumbnail-provider-plan.md`.

## 5. Современные колонки `.fb2` в Проводнике

### Почему старый ColumnProvider не подходит

Старый `IColumnProvider` — это не та база, на которую стоит опираться для
современных Windows. Даже если его частично оживить, цена поддержки будет
слишком высокой, а результат — сомнительным.

Для современных колонок нужен новый контур свойств файла, а не реанимация
старого `ColumnProvider`.

### Целевой подход

Нужно проектировать современный shell property handler для `.fb2`, который:

- читает метаданные FB2 без запуска FBE;
- отдаёт их Проводнику через современную property-модель Windows;
- позволяет использовать эти значения и в tooltip, и в columns/details, и в
  поиске/индексации там, где это поддерживается системой.

### Рекомендуемый MVP по свойствам

На первом этапе не нужно пытаться покрыть всё. Достаточно 4-6 полей:

1. `Автор`;
2. `Название`;
3. `Жанр`;
4. `Язык`;
5. `Серия`;
6. `Версия документа` или `Дата документа`.

Этого уже достаточно, чтобы пользователь увидел реальную пользу в Проводнике.

### Источник данных

Метаданные можно брать из уже знакомых FBE структур FB2:

- `description/title-info`;
- `description/document-info`.

Логично переиспользовать существующую SAX-логику разбора FB2 как reference, но
не тащить старый `ColumnProvider` целиком в новую архитектуру.

### Этапы внедрения современных колонок

#### Этап 1. Исследование и проектирование

- определить минимальный набор свойств;
- сопоставить свойства FB2 со свойствами Windows Shell;
- решить, какие свойства маппятся на стандартные Windows keys, а какие требуют
  отдельной схемы.

Промежуточное правило для текущего кода:

- `Author`, `Title`, `Language` остаются стандартными Windows properties;
- `Keywords`, `Genre`, `Sequence`, `DocumentId`, `DocumentVersion`,
  `DocumentDate` переведены в отдельные FBE-specific свойства.

Для FBE-specific полей выбран единый дальнейший путь:

- не временные legacy-обходы;
- а собственная FBE-specific property schema с явным префиксом `FBE:` в
  отображаемых названиях колонок.

Отдельный рабочий план вынесен в:

- `docs/sequence-property-schema-plan.md`.

Отдельная сводная таблица решений по свойствам лежит в
`docs/fb2-property-mapping.md`.

Желательно, чтобы это правило было зафиксировано не только в документации, но
и прямо в коде внутреннего слоя свойств: future property handler должен
опираться на явные признаки вроде "использует стандартное Windows-свойство",
"подтверждено" и "входит в MVP", а не на разбор строк руками.

#### Этап 2. Выделение ядра чтения метаданных

- вынести безопасный reader метаданных `.fb2` в отдельный внутренний модуль;
- сделать его независимым от legacy shell API;
- покрыть его небольшим набором regression-тестов на реальных `.fb2`.

Промежуточно сюда уже можно отнести и отдельный внутренний слой маппинга
`PropertyId -> PROPERTYKEY` для подтверждённых MVP-полей. Это полезно, чтобы
будущий property handler работал не со строками canonical name, а сразу с
типобезопасными Windows-ключами свойств.

Следующий безопасный шаг после этого — внутренний кэш значений будущего
`IPropertyStore`: слой, который уже умеет хранить и отдавать
`PROPERTYKEY + PROPVARIANT`, но пока ещё не является COM-объектом shell.

После него разумный промежуточный этап — тестируемый COM-каркас
`IPropertyStore + IInitializeWithStream` без регистрации в системе и без
подключения к Explorer. Это позволяет проверить форму будущего handler-а,
контракт чтения из `IStream` и read-only поведение ещё до shell-интеграции.

Ещё один полезный переходный слой после этого — отдельный класс
`Fb2PropertyHandler`, который уже выглядит как будущая точка входа modern
shell handler, но пока не включён в реальную регистрацию и не подключён к
поставке по умолчанию.

Чтобы не смешивать эксперимент с обычной поставкой, полезно держать его за
отдельным compile-time флагом. Тогда modern handler можно включать только в
специальной сборке для ручной проверки, не меняя поведение стандартного
релизного контура.

Следом полезно подготовить отдельные операционные скрипты:

- регистрация experimental handler;
- снятие регистрации;
- краткий GUI-чеклист проверки.

Тогда ручное тестирование становится воспроизводимым и не зависит от
разрозненных ручных действий.

Для самой ассоциации experimental handler лучше опираться на официальную схему
регистрации property handlers из Microsoft Learn: CLSID обработчика плюс
ассоциация расширения файла через ветку `PropertySystem\PropertyHandlers`.

#### Этап 3. Прототип property handler

- собрать минимальный shell-компонент только для чтения свойств;
- проверить отображение 2-3 полей в Details/columns на Windows 10/11;
- убедиться, что нет падений `explorer.exe` на битых или больших FB2.

Промежуточный фактический статус:

- отдельные `Win32`- и `x64`-варианты modern property handler уже включены в
  релизный контур;
- в упаковке они разведены по именам:
  - `FBShell.dll` для `Win32 Explorer`;
  - `FBShell64.dll` для `64-bit Explorer`;
- значения `.fb2` уже подтверждённо отображаются в колонках Проводника;
- FBE-specific свойства `Genre`, `Sequence`, `DocumentVersion`,
  `DocumentDate` уже публикуются через отдельную schema;
- shell-строки `InfoTip`, `TileInfo`, `Details` для `FictionBook.2` уже
  возвращены в актуальный NSIS-контур без возврата legacy `ColumnProvider`;
- дополнительно подтверждено, что для правой панели сведений Windows 11 нужен
  отдельный ключ `PreviewDetails`; после его добавления правая панель тоже
  начала показывать значения из `.fb2`;
- следующий практический шаг теперь уже не в базовой shell-регистрации, а в
  дальнейшей ревизии `IconExtractor` и legacy shell-контуров.

#### Этап 4. Расширение набора свойств

- добавить жанр, серию, язык и дополнительные поля;
- проверить tooltip/details/search там, где это возможно;
- оценить производительность на папках с большим числом FB2.

#### Этап 5. Упаковка и режим установки

- сделать установку modern property handler отдельной и явной;
- не смешивать её с legacy `ColumnProvider`;
- предусмотреть сценарий отключения или удаления без ручной чистки реестра.

## 6. Предлагаемый порядок разработки

Лучший практический порядок такой:

1. вывести `ColumnProvider` из стандартной поставки;
2. подготовить отдельный reader метаданных FB2;
3. решить задачу `Validate` отдельно от колонок;
4. после этого развивать modern property handler для колонок и published-
   свойств, не завязывая план на tooltip Windows 11;
5. только в конце решать, нужен ли обновлённый `IconExtractor`.

Это уменьшает риск и не смешивает три разных направления в одну большую
непрозрачную задачу.

## 7. Что можно сделать ближайшими шагами

### Ближайший шаг 1

Дальше упростить и документально закрепить legacy-контур:

- не возвращать `ColumnProvider`, `ContextMenu` и `IconExtractor` в активный
  проект без отдельной причины;
- держать стандартную поставку только на modern property handler и обычной
  shell-команде `Validate`.

### Ближайший шаг 2

Подготовить техническую основу для современных колонок:

- создать внутреннюю модель метаданных `.fb2`;
- вынести безопасный parser title/document-info;
- добавить тестовые `.fb2` и regression-проверки parser-а.

### Ближайший шаг 3

После этого уже можно начинать экспериментальный modern property-handler
прототип для 2-3 полей.

## 8. Практический вывод

Переработка `FBShell.dll` возможна, но как серия небольших шагов, а не как
"обновление одной старой DLL".

Современные колонки `.fb2` в Проводнике тоже реалистичны, но это должна быть
новая архитектура поверх современного shell property handler, а не попытка
реанимировать старый `IColumnProvider`.
