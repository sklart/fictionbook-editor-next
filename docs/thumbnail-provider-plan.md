# План modern thumbnail provider для `.fb2`

Этот документ фиксирует следующий этап после завершения modern property
handler: безопасное возвращение обложек `.fb2` в Проводник без оживления
legacy `IconExtractor`.

## 1. Исходная точка

Старый `IconExtractor` сейчас нам не подходит как основа для возврата в
стандартную поставку:

- он реализован через legacy `IExtractImage` / `IExtractImage2`;
- он работает прямо внутри `explorer.exe`;
- он содержит собственный SAX-разбор `coverpage` и ручной base64-декодер;
- он тянет древний bundled image-stack:
  - `zlib 1.1.4`;
  - `libpng 1.2.8`;
  - `libjpeg 6b`.

Практически это значит, что прямое возвращение старого кода дало бы красивую
функцию, но плохой профиль риска.

## 2. Целевой подход

Следующий контур нужно строить не как revival старого `IconExtractor`, а как
новый modern thumbnail provider.

Рабочая идея на текущем этапе такая:

- отдельный COM-класс thumbnail provider;
- современная инициализация через `IInitializeWithStream`;
- чтение `.fb2` из потока тем же безопасным способом, что уже используется в
  modern property handler;
- извлечение `coverpage` в отдельный внутренний reader, не завязанный на
  legacy shell API;
- декодирование изображения через системный стек Windows, а не через старые
  встроенные `zlib/libpng/libjpeg`.

Важно: это пока технический план на базе стандартной современной shell-модели
Windows, а не уже готовый зарегистрированный контур.

## 3. Что уже можно переиспользовать

У нас уже есть полезная основа из modern property handler:

- `src\fbe\Fb2Metadata.*` как безопасная точка входа в чтение `.fb2`;
- `src\fbshell\Fb2PropertyStoreCom.*` как рабочий COM-каркас с
  `IInitializeWithStream`;
- готовый сценарий сохранения входного `IStream` во временный файл;
- отдельный релизный и тестовый контур для `FBShell.dll` / `FBShell64.dll`;
- наработанные скрипты регистрации / снятия регистрации experimental-компонентов.

То есть для thumbnail provider не нужно снова изобретать весь operational
контур с нуля.

## 4. Что нужно сделать отдельно

### Этап 1. Выделить reader обложки

Нужен отдельный внутренний модуль, который:

- читает `title-info/coverpage/image/@href`;
- находит соответствующий `binary` по `id`;
- возвращает:
  - MIME-тип;
  - сырые байты изображения;
  - статус ошибки.

На этом шаге нельзя тащить старый `IconExtractor.cpp` как есть. Его можно
использовать только как reference по структуре данных FB2.

### Этап 2. Подготовить smoke-проверку reader-а

Нужен отдельный тестовый `.fb2` с обложкой и маленький smoke-runner, который
проверяет:

- что обложка вообще находится;
- что MIME-тип читается корректно;
- что байты не пустые;
- что битый или неполный `coverpage` не валит обработчик.

### Этап 3. Выбрать современный image-decoder

Цель этапа — отказаться от legacy `ImageLoader + zlib/libpng/libjpeg`.

Минимальный практический путь:

- брать байты обложки;
- декодировать их через системный стек Windows;
- на выходе получать bitmap/thumbnail нужного размера.

На этом шаге важно специально проверить:

- PNG;
- JPEG / JPG;
- BMP;
- отсутствие обложки;
- слишком большую картинку;
- битое изображение.

### Этап 4. Собрать experimental thumbnail provider

После reader-а и декодера можно делать отдельный experimental COM-класс:

- не включённый в стандартную поставку;
- с собственной регистрацией;
- без смешивания с current property handler.

MVP для этого этапа:

- Проводник получает thumbnail для `.fb2` с обложкой;
- файлы без обложки спокойно падают в обычную иконку;
- `explorer.exe` не падает на битых книгах.

### Этап 5. Ручная GUI-проверка

Для ручной проверки потом нужен отдельный чеклист:

- папка с несколькими `.fb2`;
- книга с JPEG-обложкой;
- книга с BMP-обложкой;
- книга с PNG-обложкой;
- книга без `coverpage`;
- битый `.fb2`;
- проверка `x86` и `x64` Проводника там, где это применимо.

## 5. Что считаем успехом

Первым успехом считается не «полностью заменить иконки в релизе», а более
скромный результат:

1. внутренний reader обложки работает и покрыт smoke-тестом;
2. отдельный experimental thumbnail provider умеет отдать thumbnail хотя бы
   для валидного `.fb2` с PNG-, JPEG/JPG- и BMP-обложкой;
3. файлы без обложки или с битыми данными не валят Проводник;
4. legacy `IconExtractor` при этом по-прежнему не возвращается в стандартную
   сборку.

## 6. Что пока сознательно не делаем

На первом проходе не стоит сразу тянуть:

- полную замену обычной иконки файла;
- кэширование поверх системного thumbnail cache;
- поддержку всех экзотических image MIME-типов;
- смешивание thumbnail provider и property handler в один обязательный
  релизный контур.

Сначала нужен узкий и проверяемый experimental путь.

## 7. Что уже выяснили на практике

В ходе ручной и полуавтоматической проверки на Windows 11 уточнилось важное
поведение shell thumbnail cache:

- прямой вызов `FBShell.dll` через наш loader-smoke может успешно читать
  встроенную обложку проблемного `.fb2`;
- принудительный вызов через `IThumbnailCache` с
  `WTS_FORCEEXTRACTION | WTS_EXTRACTINPROC` тоже может успешно вернуть
  миниатюру;
- при этом обычный путь `IShellItemImageFactory::GetImage` для того же файла
  иногда продолжает возвращать старый `0x8004B200`.

Практически это означает, что часть проблемных книг упирается не в reader
обложки и не в COM-регистрацию thumbnail provider, а в stale negative cache
самого shell-контура Windows.

Из этого появился отдельный сервисный обходной сценарий:

- `tools/build/prime-fb2-thumbnail-cache.ps1`

Он принудительно перепрожигает системный thumbnail cache для выбранных `.fb2`,
когда обычный Проводник уже застрял на старом пустом результате.

Отдельно уже подтверждена ещё одна практическая причина сбоев:

- ранняя версия provider возвращала полноразмерный `HBITMAP` обложки,
  игнорируя запрошенный shell-ом `cx`;
- для части книг in-proc путь это терпел, а out-of-proc shell-thumbnail
  pipeline падал уже после успешного `GetThumbnail`;
- после перевода provider на явное масштабирование до requested `cx`
  обычный `IShellItemImageFactory` для проблемных книг начал проходить.
