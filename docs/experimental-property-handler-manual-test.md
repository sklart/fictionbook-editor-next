# Ручная проверка experimental property handler

Этот документ нужен для управляемой GUI-проверки нового modern property
handler для `.fb2`.

## Перед началом

Нужно понимать две вещи:

1. это экспериментальный контур;
2. он не должен проверяться поверх случайного состояния старых shell-настроек.

Поэтому перед проверкой лучше:

- закрыть лишние окна Проводника;
- понимать, какая именно сборка лежит в `out\Release`;
- иметь под рукой тестовый `.fb2`.

Подходящий файл уже есть в проекте:

- `tools/tests/fb2-metadata-smoke.fb2`

## Подготовка

Оба скрипта регистрации требуют запуска из PowerShell с правами администратора.

Собрать и зарегистрировать experimental handler:

```powershell
.\tools\build\register-experimental-property-handler.ps1 -Configuration Release
```

Если нужно отдельно проверить FBE-specific свойства (`FBE:Ключевые слова`,
`FBE:Жанр`, `FBE:Серия`, `FBE:Идентификатор документа`,
`FBE:Версия документа`, `FBE:Дата документа`), до GUI-проверки дополнительно
зарегистрировать FBE-specific property schema:

```powershell
.\tools\build\register-sequence-property-schema.ps1
```

Быстро посмотреть текущее состояние published FBE-specific свойств можно и без
прав администратора:

```powershell
.\tools\tests\check-fbe-specific-properties.ps1
```

Этот скрипт теперь дополнительно показывает и shell-строки класса
`FictionBook.2`:

- `InfoTip`
- `TileInfo`
- `Details`
- `PreviewDetails`

То есть перед GUI-проверкой им можно быстро увидеть не только published
`FBE.*`-свойства, но и то, прописаны ли в системе строки, от которых зависят
tooltip/плитки/сведения.

Откатить регистрацию после проверки:

```powershell
.\tools\build\unregister-experimental-property-handler.ps1 -Configuration Release
```

## Что именно проверять

### 1. Базовый сценарий

Открыть папку с `.fb2` и убедиться, что Проводник не падает и не зависает.

### 2. Колонки Details

Попробовать добавить и проверить поля:

- `Автор`
- `Название`
- `Язык`
- `FBE:Ключевые слова`
- `FBE:Жанр`
- `FBE:Идентификатор документа`
- `FBE:Версия документа`
- `FBE:Дата документа`
- `FBE:Серия`

Ожидаемый смысл:

- `Автор` должен показывать автора книги;
- `Название` — название книги;
- `Язык` — язык из `title-info/lang`.
- `FBE:Ключевые слова` — содержимое `title-info/keywords`, если оно есть;
- `FBE:Жанр` должен показывать книжные жанры из `title-info/genre`;
- `FBE:Идентификатор документа` — значение `document-info/id`;
- `FBE:Версия документа` — значение `document-info/version`;
- `FBE:Дата документа` — дату FB2-документа;
- `FBE:Серия` должна показывать значение `title-info/sequence`, если заранее
  зарегистрирована отдельная FBE-specific property schema.

Важно: стандартные Windows-поля должны называться по локализации Windows, а
FBE-specific поля намеренно показываются с префиксом `FBE:`.

### 3. Tooltip / сведения

Навести курсор на `.fb2` и проверить, появились ли новые свойства в tooltip
или в области сведений.

Текущий подтверждённый результат на Windows 11 такой:

- колонки FBE-specific свойств отображаются и читают корректные значения;
- после полной корректной shell-регистрации tooltip начинает показывать не
  только базовые поля, но и данные из `.fb2`, включая `Авторы`, `Название`,
  `Язык`, `FBE:Серия`, `FBE:Версия документа`, `FBE:Дата документа`;
- после добавления shell-строки `PreviewDetails` правая панель сведений тоже
  начинает показывать данные из `.fb2`, включая `Авторы`, `Название`, `Язык`,
  `FBE:Жанр`, `FBE:Серия`, `FBE:Версия документа`, `FBE:Дата документа`,
  `FBE:Ключевые слова`, `FBE:Идентификатор документа`.

То есть для текущего published-контура на Windows 11 уже считаются рабочими и
колонки, и tooltip, и правая панель сведений, если shell-регистрация
действительно полная и включает `PreviewDetails`.

### 4. Устойчивость на наборе файлов

Проверить:

- валидный `.fb2`;
- большой `.fb2`;
- битый `.fb2`, если будет подготовлен отдельно.

Новый контур не должен валить `explorer.exe`.

### 5. Откат

После проверки обязательно снять регистрацию:

```powershell
.\tools\build\unregister-experimental-property-handler.ps1 -Configuration Release
```

Если отдельно регистрировалась schema для `Sequence`, её тоже нужно снять:

```powershell
.\tools\build\unregister-sequence-property-schema.ps1
```

## Что фиксировать по результатам

По каждой проверке полезно записать:

- видит ли Проводник новые поля;
- какие именно поля реально доступны;
- совпадают ли значения с ожиданием;
- как выглядят названия полей в русской Windows;
- появляется ли что-то в tooltip/details;
- есть ли задержки, зависания, перезапуск `explorer.exe` или ошибки.

## Что сейчас считается ожидаемым результатом

Для текущего published-контура мы уже целимся как минимум в такие свойства:

- `System.Author`
- `System.Title`
- `System.Language`

После отдельной регистрации FBE-specific schema дополнительно должен стать
доступен published-набор ключей:

- `FBE.Keywords`
- `FBE.Genre`
- `FBE.Sequence`
- `FBE.DocumentId`
- `FBE.DocumentVersion`
- `FBE.DocumentDate`

Если Explorer стабильно видит эти поля и читает по ним значения из `.fb2`, это
подтверждает, что modern property handler уже закрывает не только базовый MVP,
но и следующий слой метаданных.

Если после корректной регистрации этих полей нет именно в правой панели
сведений, первым делом стоит проверить наличие shell-строки `PreviewDetails`,
а уже потом подозревать ограничения конкретной сборки Windows.
